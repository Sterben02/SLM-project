## 1. Контекст и актуальность

Утечки секретов (API-ключей, токенов, приватных ключей) — одна из самых частых причин инцидентов информационной безопасности в современных software-проектах. По данным исследований, более 60% утечек происходят из-за секретов, случайно оставленных в исходном коде, коммитах или конфигурационных файлах.

Параллельно с секретами, в коде регулярно встречаются **небезопасные паттерны**: использование `eval()`, SQL-конкатенация, вызовы shell с `shell=True`, слабые хэши, отключённая TLS-проверка. Эти фрагменты не являются секретами, но создают уязвимости (RCE, SQLi, MITM и др.).

Для борьбы с этими проблемами используются **SAST-сканеры** (Static Application Security Testing) — инструменты статического анализа кода.

## 2. Проблема

Существующие сканеры секретов (gitleaks, detect-secrets, TruffleHog, GitHub Secret Scanning) опираются преимущественно на три метода:

1. **Регулярные выражения** — поиск строк, соответствующих известным форматам (например, `AKIA[0-9A-Z]{16}` для AWS).
2. **Энтропия Шеннона** — поиск «случайных» строк с высокой информационной энтропией.
3. **Контекстные ключевые слова** — анализ имён переменных (`api_key`, `password`, `token`).

Эти методы работают быстро, но имеют **системный недостаток — высокий уровень ложных срабатываний (False Positive Rate)**:

- тестовые значения (`API_KEY = "test_key_123"`);
- примеры из документации и README;
- UUID, хэши, base64-код, случайные идентификаторы;
- закомментированные секреты;
- строки, похожие на секреты, но таковыми не являющиеся.

На практике FPR классических сканеров может достигать 30-60%, что приводит к «алерт-усталости» у разработчиков и игнорированию реальных угроз.

Кроме того, классические SAST-инструменты плохо справляются с **семантическим анализом**: они не учитывают смысл кода, назначение файла, комментарии и соседние строки.

## 3. Гипотеза

**Small Language Model (SLM)** — небольшая языковая модель (1-7B параметров), работающая локально, — способна учитывать контекст фрагмента кода (имя переменной, окружающие строки, комментарии, тип файла) и за счёт этого:

1. **Снизить FPR** при поиске секретов, отличая реальные секреты от тестовых значений, примеров и случайных идентификаторов.
2. **Обнаруживать небезопасные паттерны кода**, которые сложно описать регулярными выражениями (например, SQL-конкатенация в разных формах).
3. **Формировать объяснения** срабатываний на естественном языке, что повышает доверие разработчиков к результатам сканирования.

Гипотеза проверяется экспериментально: сравниваются метрики классических baseline-детекторов и SLM-детектора на едином датасете.

## 4. Цели проекта

Разработать прототип CLI-сканера, который:

1. Сканирует локальный каталог с исходным кодом.
2. Обнаруживает секреты 5 классов: API-ключи, access-токены, пароли, приватные ключи, JWT.
3. Обнаруживает небезопасные фрагменты 6 классов: `eval`, `exec`, `shell=True`, SQL-конкатенация, слабые хэши MD5/SHA1, захардкоженные учётные данные.
4. Сочетает классические baseline-детекторы (regex, entropy, keyword) с SLM-детектором.
5. Формирует отчёт в форматах JSON и Markdown.
6. Для каждого срабатывания предоставляет объяснение — почему фрагмент считается подозрительным.
7. Сравнивает метрики baseline и SLM-подходов.

## 5. Границы проекта (Scope)

**Входит в проект:**
- Анализ исходного кода на Python, JavaScript, Go, Java (базовая поддержка).
- Поиск секретов и небезопасных паттернов в файлах.
- Локальный запуск SLM (без обращения к внешним API).
- Синтетический + частично open-source датасет для обучения/оценки.

**Не входит в проект:**
- Анализ git-истории (commits, diffs).
- Анализ бинарных файлов, скомпилированного кода.
- Сканирование зависимостей (SCA).
- Динамический анализ (DAST).
- Интеграция с CI/CD-системами (только локальный запуск).
- Поддержка всех существующих языков программирования.

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| SLM слишком медленная для больших репозиториев | Средняя | Высокое | Двухступенчатая фильтрация: baseline отбирает кандидатов, SLM проверяет только их |
| Сложно собрать качественный размеченный датасет | Высокая | Высокое | Генерация синтетики + использование открытых датасетов |
| SLM даёт галлюцинации в объяснениях | Средняя | Среднее | Шаблонные объяснения + проверка по фактам из кода |
| Ложные срабатывания на специфичных проектах | Средняя | Среднее | Возможность тонкой настройки правил и white-list |
